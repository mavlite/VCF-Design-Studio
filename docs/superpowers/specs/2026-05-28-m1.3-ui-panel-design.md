# M1.3 Gateway Interface UI Panel — Design

**Date:** 2026-05-28
**Branch:** `feat/m1.3-ui-panel`
**Context:** PR #104 landed the M1.3 foundation (model surface, cell-map, migration, 17 tests). The ClusterCard UI for editing the new fields is the remaining piece per the project's standing rule ("Ship UI for new models"). This spec captures the agreed-on UI design before implementation.

## Goal

Let the user edit, from the ClusterCard panel, the M1.3 model fields that today require JSON import:

- `cluster.networks.uplinks[0..1].{vlan, gateway}` — two T0 uplinks, shared across edge nodes
- `cluster.edgeCluster.nodes[0..1].gatewayInterfaceIps[0..1]` — per-edge-node uplink IPs

## Placement

Two additions inside [vcf-design-studio-v9.jsx](../../../vcf-design-studio-v9.jsx), co-located with related controls (chosen over a single dedicated panel for data adjacency):

1. **T0 Uplinks sub-section** in `ClusterCard`, inserted right after the existing T0 Gateways section (~L1740). Edits `cluster.networks.uplinks[]`.
2. **Per-node Gateway Interface IPs row** in `EdgeClusterPanel`, inserted inside each Edge Node block right after the existing per-node Uplinks / TEP IPs rows (~L3434–L3459). Edits `cluster.edgeCluster.nodes[j].gatewayInterfaceIps[]`.

## Gating

- **T0 Uplinks sub-section:** rendered only when `cluster.t0Gateways.length > 0`. T0 Uplink VLAN/Gateway are meaningless until at least one T0 Gateway has been defined; hiding the sub-section avoids prompting the user for values they can't act on. When `t0Gateways` is empty, the fields remain at their factory defaults (`{vlan: null, gateway: ""}`) and emit nothing.
- **Per-node Gateway Interface IPs:** unconditional inside each Edge Node block. The `EdgeClusterPanel` itself is the implicit gate (renders unconditionally inside `ClusterCard` today); within an edge node block, hiding these rows would be inconsistent with the surrounding per-node fields.

|---|---|---|---|---|
| T0 Uplink 1 VLAN | `cluster.networks.uplinks[0].vlan` | `number \| null` | `<input type="number" min="0" max="4094">` | Non-numeric → `null` (matches existing model contract) |
| T0 Uplink 1 Gateway | `cluster.networks.uplinks[0].gateway` | `string` | `<input type="text">` IPv4 placeholder `10.x.x.1` | Passthrough string |
| T0 Uplink 2 VLAN | `cluster.networks.uplinks[1].vlan` | `number \| null` | same shape as Uplink 1 VLAN | same |
| T0 Uplink 2 Gateway | `cluster.networks.uplinks[1].gateway` | `string` | same shape as Uplink 1 Gateway | same |
| Edge Node N Uplink 1 IP | `cluster.edgeCluster.nodes[N-1].gatewayInterfaceIps[0]` | `string` | `<input type="text">` CIDR placeholder `10.x.x.2/24` | Passthrough string |
| Edge Node N Uplink 2 IP | `cluster.edgeCluster.nodes[N-1].gatewayInterfaceIps[1]` | `string` | same | same |

## Visual idiom

**T0 Uplinks sub-section** — matches the visual style of the immediately-adjacent T0 Gateways section in `ClusterCard` for cohesion. Two-column grid (`grid grid-cols-1 lg:grid-cols-2 gap-2`), each cell containing a small card with the same border/padding/font-mono treatment used elsewhere in `ClusterCard`. Card header: `T0 Uplink 1` / `T0 Uplink 2` in `text-[10px] uppercase tracking-[0.18em]` (existing internal heading convention). Two stacked label+input rows per card: `VLAN` (small width) and `Gateway` (flex-1).

**Per-node Gateway Interface IPs row** — matches the per-node Uplinks/TEP IPs row shape inside [EdgeClusterPanel:3434-3459](../../../vcf-design-studio-v9.jsx#L3434-L3459). A labeled `Gateway Interface IPs` row with two `<input>` controls side-by-side (`grid grid-cols-2 gap-2`), each labeled `Uplink 1 IP` / `Uplink 2 IP`. Same input className the panel already uses: `text-xs font-mono bg-white border border-slate-200 rounded px-2 py-1.5 w-full text-slate-700`.

## State update pattern

Both sections follow the existing `ClusterCard` / `EdgeClusterPanel` pattern: build a new array via `map` on edit, dispatch via the component's existing `update` / equivalent handler.

```js
// T0 Uplinks (inside ClusterCard)
const updateUplink = (i, patch) => {
  const uplinks = (cluster.networks?.uplinks ?? []).map(
    (u, idx) => idx === i ? { ...u, ...patch } : u
  );
  update({ networks: { ...cluster.networks, uplinks } });
};

// Edge Node Gateway Interface IPs (inside EdgeClusterPanel, per node)
const updateNodeIp = (nodeIdx, ipIdx, value) => {
  const nodes = cluster.edgeCluster.nodes.map((n, j) => {
    if (j !== nodeIdx) return n;
    const ips = (n.gatewayInterfaceIps ?? ["", ""]).slice();
    ips[ipIdx] = value;
    return { ...n, gatewayInterfaceIps: ips };
  });
  update({ edgeCluster: { ...cluster.edgeCluster, nodes } });
};
```

Handler names should mirror the immediate neighbor (e.g., the T0 Gateways section's existing update helper) rather than introducing a new convention. This is a styling detail, not a requirement.

## Tests

Two new files under [tests/unit/components/](../../../tests/unit/components/) using the M2.2 JSDOM + RTL stack (each file opts in via `// @vitest-environment jsdom` pragma).

### `m1.3-t0-uplinks-panel.test.jsx`

| Case | Assertion |
|---|---|
| 1 | Sub-section hidden when `cluster.t0Gateways.length === 0` |
| 2 | Sub-section visible when `cluster.t0Gateways.length > 0` |
| 3 | Typing a VLAN updates `cluster.networks.uplinks[0].vlan` |
| 4 | Typing a Gateway updates `cluster.networks.uplinks[0].gateway` |
| 5 | Non-numeric VLAN coerces to `null` in dispatched patch |

### `m1.3-edge-node-gateway-ips.test.jsx`

| Case | Assertion |
|---|---|
| 1 | Both Edge Node blocks render the Gateway Interface IPs row |
| 2 | Typing in Node 1 Uplink 1 IP updates `nodes[0].gatewayInterfaceIps[0]` |
| 3 | Typing in Node 2 Uplink 2 IP updates `nodes[1].gatewayInterfaceIps[1]` only |
| 4 | Independent state: editing one node does not mutate the other |

No round-trip / cell-map tests — already covered exhaustively by [theme-m1.3-gateway-interface.test.js](../../../tests/unit/themes/theme-m1.3-gateway-interface.test.js).

## Out of scope

- **New validation rules** (VCF-IP-XXX). Format-level input only; cross-field validation (uplink VLAN vs TEP VLAN distinctness, gateway-in-subnet, CIDR mask sanity) is deferred. Add only if architect flags a real risk.
- **M1.4 sub-network MTU/Range/Type fields.** Tracked separately; needs model expansion.
- **Theme 19 AZ2 mirror.** AZ2 doesn't carry uplinks; not applicable.
- **9.1 vDPG flag (M1.5b).** Tracked separately; needs new fleet-level field.
