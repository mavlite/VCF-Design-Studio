# M2.1: Round-trip Coverage Matrix — Design

**Date:** 2026-05-29
**Branch:** `feat/m2.1-round-trip-matrix`
**Context:** Round-trip coverage today is partial and ad-hoc. `tests/unit/round-trip-depth.test.js` exercises *every value-bearing field* of supervisor / witness / federation config (themes 11/12/13) through a 9.1 CSV cell-map round-trip using unique sentinels; `tests/unit/themes/theme-19-az2-networking.test.js` does an emit→xlsx→parse→import round-trip for AZ2 stamps; `tests/unit/themes/theme-10-vcf-network-pools.test.js` does CSV-level round-trips. But most of the model surface has no systematic round-trip assertion, and there is no guard that *flags* a newly-added field lacking coverage.

## Why this matters

This codebase's most common silent-regression class is a field that stops surviving a round-trip:

- **Workbook (CSV cell-map) path** — a `resolve`/`apply` pairing bug in `WORKBOOK_CELL_MAP` (or a missed cell-map entry) drops a field on `emit → import`. The user's design exports and re-imports subtly wrong, with no error.
- **JSON save/load path** — `migrateFleet` whitelist-merges on import. A field added to a factory's output but not reflected in the migration whitelist is silently stripped when a user saves a design to JSON and reloads it.

Both are invisible today unless a hand-written test happens to assert the specific field. The operating-convention memory already says *"every exported field must have a UI editor path; audit before any export PR"* — M2.1 makes the **round-trip half** of that an automated, complete invariant.

## Goal

A **complete, self-guarding round-trip matrix**: every value-bearing model field is asserted to survive whichever round-trip path(s) apply, and a meta-guard fails CI when a new field escapes coverage.

Concretely, after this work:
1. A single kitchen-sink fleet is the canonical "model surface" — every factory invoked, every value-bearing scalar set to a unique, type-aware sentinel.
2. Every sentinel survives the **JSON** round-trip (guards `migrateFleet` whitelist completeness).
3. Every **workbook-mapped** sentinel survives the **CSV cell-map** round-trip on both 9.0 and 9.1.
4. A **meta-guard** asserts `leaves == CSV-matrix-paths ∪ non-workbook-allowlist ∪ structural-ignore`; a new uncovered field fails the guard, naming the orphan path.

## Non-goals

- Not testing the `.xlsx` binary emit/read path field-by-field — `theme-19` + `workbook-xlsx-emitter.test.js` cover the xlsx serialization layer; the CSV cell-map path exercises the same `resolve`/`apply` logic without the binary round-trip cost.
- Not asserting UI editability (that is the other half of the export-coverage convention; out of scope here).
- Not changing `engine.js` behavior. If the matrix surfaces a real round-trip bug, that fix is a separate, clearly-scoped change.

## Components

### 1. `tests/helpers/kitchen-sink-fleet.js` (new; not a `.test` file)

Builds a maximally-populated fleet and returns `{ fleet, sentinels }`.

- **Coverage:** every `create*` / `new*` factory is invoked at least once in a reachable position (fleet-level configs; ≥1 instance; mgmt + workload domains; ≥1 cluster per domain with networks, az2Networks, az2HostOverlay, portgroups, vds, supervisorConfig, nsxHostOverlay, edge cluster + nodes, t0 gateways, host overrides; federation global/local/tier1; witness; AD; backup; installer; naming; report metadata).
- **Sentinels are type-aware** (the key implementation cost):
  - strings → unique token `"rt::<dot.path>"`
  - numbers (counts, sizes) → distinct in-range integers
  - VLAN ids → distinct values within `[VLAN_ID_MIN, VLAN_ID_MAX]`
  - IPs / CIDRs / pools → distinct valid addresses (the cell-map validates octet ranges + CIDR form on apply, so invalid sentinels would be rejected)
  - enums / dropdowns → a valid member (recorded as the expected value)
  - booleans → a value distinct from the factory default where possible
- `sentinels` is a flat map `"dot.path" → value` covering exactly the value-bearing leaves the helper set.
- Version: produces a **9.1 superset** by default (9.1 adds fields over 9.0); a `vcfVersion: "9.0"` option yields the 9.0-valid variant for version-specific assertions.
- **Self-check:** `sizeFleet(fleet)` must run without throwing on the kitchen-sink (proves the populated fleet is internally coherent).

### 2. `tests/unit/round-trip-matrix.test.js` (new)

Three layers, all driven by the shared kitchen-sink:

- **JSON completeness layer** — `migrateFleet(JSON.parse(JSON.stringify(fleet)))`; assert every `sentinels` path deep-equals its expected value on the rebuilt fleet. One failing path = a `migrateFleet` whitelist gap.
- **CSV cell-map layer** — for each of `9.0`, `9.1`: `emitWorkbookCellMapCsv → parseWorkbookCellMap → importWorkbookCellMap`; assert every **workbook-mapped** sentinel path survives. Mapped paths are an explicit, reviewable list (the matrix), grouped by theme via `describe` blocks for navigability.
- **Meta-guard layer** — recursively walk the kitchen-sink `fleet` to enumerate **all** leaf paths, partition them, and assert:
  - `all-leaves == value-bearing-leaves ∪ STRUCTURAL_IGNORE`, where value-bearing-leaves `== CSV-matrix-paths ∪ NON_WORKBOOK_ALLOWLIST`.
  - every **value-bearing** leaf (i.e. every `sentinels` path) survives the JSON round-trip. Structural leaves are *not* asserted to survive — `migrateFleet` may regenerate `id`s and re-derive `type`, so they carry no user data and are deliberately excluded.
  - On mismatch, fail with the orphan paths listed and the instruction: *"add to the CSV matrix, NON_WORKBOOK_ALLOWLIST, or STRUCTURAL_IGNORE."*

  `NON_WORKBOOK_ALLOWLIST` — value-bearing fields that legitimately do not stamp to the workbook (each entry carries a `// why:`): formula-derived cells the studio intentionally does not stamp (NSX GM Node 3 search list, AZ2 T0 BGP slots 3+4), UI-only/computed fields, etc.
  `STRUCTURAL_IGNORE` — non-value leaves: `id`, `type`, `isDefault`, array-index scaffolding, and other structural keys that carry no user data.

### 3. Disposition of `round-trip-depth.test.js`

Fold its themes 11/12/13 sentinel assertions into the matrix so there is **one** matrix and the meta-guard sees those paths as covered. Delete `round-trip-depth.test.js` once its assertions are represented in the matrix (no coverage lost — the matrix is a superset).

## Data flow

```
kitchen-sink-fleet.js  ──►  { fleet, sentinels }
        │
        ├─► JSON round-trip (stringify→parse→migrateFleet) ──► assert ALL sentinels survive
        ├─► CSV round-trip 9.0 (emit→parse→import) ─────────► assert CSV-matrix sentinels survive
        ├─► CSV round-trip 9.1 (emit→parse→import) ─────────► assert CSV-matrix sentinels survive
        └─► walk leaves ──► meta-guard: leaves == CSV-matrix ∪ allowlist ∪ structural-ignore
```

## Error handling

- Guard failures name exact orphan path(s) + remediation instruction — the test output should make it obvious where a new field must be registered.
- Sentinel mismatches report `path`, expected, and actual, so a `resolve`/`apply` bug points straight at the field.

## Testing

The matrix *is* the test. The only auxiliary check is the kitchen-sink self-check (`sizeFleet` runs clean; the helper sets every factory's value-bearing fields — verified by the meta-guard's leaf walk covering them).

## Phased implementation (becomes the plan)

1. **Phase 1 — kitchen-sink helper + JSON completeness guard.** Highest immediate value: catches any current `migrateFleet` whitelist gap across the whole model.
2. **Phase 2 — CSV cell-map matrix.** Extend sentinel round-trip assertions to every workbook-mapped field on 9.0 + 9.1; fold in `round-trip-depth`.
3. **Phase 3 — meta-guard + allowlists.** Close the loop so future fields can't escape; author `NON_WORKBOOK_ALLOWLIST` / `STRUCTURAL_IGNORE` with `// why:` reasons.

## Risks

- **Type-aware sentinel population** is the bulk of the effort: every numeric/IP/CIDR/VLAN/enum field needs a distinct *valid* value, because the cell-map validates on apply. Mitigation: a small set of typed sentinel generators keyed by field kind, and the kitchen-sink self-check (`sizeFleet`) catches incoherent values early.
- **9.0 vs 9.1 surface drift** — some fields exist only on one version. The kitchen-sink produces both variants; the CSV layer runs per version and the matrix marks version-specific paths.
- **Allowlist honesty** — an over-broad allowlist hides real gaps. Mitigation: every allowlist entry needs a `// why:` tied to one accepted category (formula-derived / UI-only / computed / structural), reviewed like the Task #31 `istanbul ignore` markers.
