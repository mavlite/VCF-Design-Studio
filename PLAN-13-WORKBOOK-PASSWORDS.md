# Plan 13 — Workbook Password Generation

> **Status:** 📋 Investigation + plan (2026-05-21). No code. Captures the design space for adding password generation to the workbook export flow so consultants don't have to hand-author 50–80 unique strong passwords per fleet deployment.
> **Goal:** Let users generate cryptographically strong, complexity-compliant passwords for every credential cell the official VCF Planning & Preparation Workbook asks for, stamp them into the exported `.xlsx`, and hand the user a vault file they can paste into their secrets manager — without the studio ever persisting a password.

Cross-references: [PLAN-11-WORKBOOK-INTEROP.md](PLAN-11-WORKBOOK-INTEROP.md) (the export/import pipeline this hooks into), [engine.js](engine.js) (`WORKBOOK_CELL_MAP`).

---

## 1. Why this work

The shipped workbook export (Plan 11 Phase 1b) stamps ~30 cells into the official Broadcom workbook. The other ~70 cells are password fields that Plan 11 explicitly skipped per the "design tool, not a secrets store" principle.

The principle was right but the conclusion was too narrow. **Generating** strong passwords on demand is materially different from **storing** them. The studio can synthesize per-credential passwords client-side at export time, stamp them into a copy of the workbook the user controls, hand the user a vault file they paste into 1Password / CyberArk / HashiCorp Vault / Bitwarden, and never persist any of it. That closes the last manual step in the workbook handoff.

Without this, every consultant has to:
1. Open a password generator 50–80 times (or worse, write a one-off bash + openssl loop)
2. Type each password into the workbook by hand (transcription errors)
3. Mentally tag each password with which cell it belongs to so they can put it in their vault later
4. Repeat the exercise for every fleet design

With this, one click produces a stamped `.xlsx` plus a structured vault file.

## 2. Inventory — what the workbook asks for

Source: cell-meta fixtures at `test-fixtures/workbook/workbook-cell-meta-{9.0,9.1}.json` (Phase 0 extraction). Numbers cited here are 9.1; 9.0 is within 5 cells either way.

### Total password-shaped cells: 78 (9.1) / 84 (9.0)

| Sheet | Cells | Examples |
|---|:---:|---|
| Deploy Management Domain | 18 | ESX Root, vCenter root/admin, NSX admin/root/audit, SDDC root, VCF Password, Ops admin, Identity Broker admin, SSH credentials |
| Configure Management Domain | 34 | NSX per-node passwords (System Root / CLI admin / CLI audit × 3 nodes), Edge root/admin, BGP peer × 4, vSAN witness root, Encryption Passphrase |
| Deploy Workload Domain | 12 | WLD vCenter root, WLD NSX admin/root/audit, SSO admin, SSO user |
| Configure Workload Domain | 8 | WLD Edge root/admin, BGP peer × 4 |
| Deploy Cluster | 6 | Cluster-add NSX/edge/BGP/vsan witness |

### Critical finding: Broadcom built in auto-generate toggles

Five cells across the workbook control whether VCF Lifecycle Manager generates the passwords itself at deploy time:

| Cell | Sheet | Label | Controls |
|---|---|---|---|
| L296 | Deploy Management Domain | "Auto-generate passwords for newly installed appliances" | Mgmt-domain appliance passwords (vCenter, NSX, SDDC, Ops, Auto) |
| D152 | Configure Management Domain | "Auto generate passwords with VCF to manage Edge node" | Mgmt Edge root/admin |
| D47 | Deploy Workload Domain | "Auto Generate Passwords" | WLD vCenter/NSX initial deploy |
| D357 | Deploy Workload Domain | "Auto-generate my passwords for newly installed appliances" | Additional WLD installs |
| D95 | Configure Workload Domain | "Auto generate passwords and manage them via VCF Ops" | WLD Edge root/admin |

**Each toggle defaults to `Unselected` in the pristine workbook.** When set to `Selected`, VCF generates and rotates the controlled passwords automatically; the user never sees or needs to provide them.

### Two camps of password cells

Splitting the 78 cells by whether the auto-generate toggles can cover them:

**Camp A — covered by auto-generate (52 cells)**
- Mgmt domain: vCenter root/admin, NSX admin/root/audit (× 3 nodes), SDDC root + VCF Password, Ops admin/root, Automation admin + SSH, Identity Broker admin
- Mgmt Edge: root + admin
- WLD: vCenter root, NSX admin/root/audit, additional-cluster equivalents
- WLD Edge: root + admin

For these, setting the toggle to `Selected` means VCF handles everything. The studio doesn't need to fill them.

**Camp B — NOT covered by auto-generate (18 cells, user MUST supply)**
| Cell(s) | Credential | Why user-required |
|---|---|---|
| L81 (mgmt) | ESX Root Password | ESXi hosts exist before VCF; root password set out-of-band during BIOS/iLO bootstrap |
| D29 (mgmt) | Encryption Passphrase | Recovery-only secret; must be kept by the customer, not generated by VCF |
| D167 / D174 / D182 / D189 (mgmt) + D110 / D117 / D125 / D132 (wld) + D459 / D464 (mgmt-additional) + D447 / D452 (deploy-cluster) | BGP Peer Password | Shared secret with the customer's physical router peers; can't be VCF-generated because the other side needs to know it |
| D403 (mgmt) + D384 (deploy-cluster) | vSAN Witness Root Password | Witness appliance lives at a third site, often outside VCF's lifecycle scope |
| D32 + D359 (wld) | SSO Administration / user Password | Often pre-existing identity-source-driven values |

For these, **the user must supply a value regardless of auto-generate state**. These are exactly the cells where consultants today open KeePass / pwgen / openssl and type something in.

## 3. Design space — three approaches

### Approach A — Default to Auto-Generate at Deploy (zero code)

- On export, stamp `Selected` into all 5 auto-generate toggles by default.
- Leave the 52 Camp A password cells blank — VCF fills them at deploy time.
- For the 18 Camp B cells, do nothing; user fills by hand or pulls from their vault.

**Pros:**
- Zero code; just additional cell-map entries that stamp `Selected`.
- Matches Broadcom's documented happy path for fleet deployments.
- Studio remains 100% credential-free.

**Cons:**
- Doesn't solve the Camp B problem (18 cells still require manual generation).
- Some orgs forbid VCF-generated passwords for compliance reasons (need pre-deployment vault entry for every credential).
- Doesn't address the user's actual request (they explicitly asked to *generate* the passwords).

**Verdict:** Worth shipping as default behavior regardless of which other approach we pick — there's no reason to leave the toggles `Unselected` when the user hasn't opted in to user-supplied creds.

### Approach B — One-shot Generate-on-Export

- Export-bar button "Generate & Export Workbook with Passwords" sits next to the existing `.xlsx` export.
- Clicking opens a modal: "Generate strong passwords for all 70 user-input password cells? They'll be stamped into the workbook and offered as a vault download. The studio will not retain any password — close the tab and they're gone."
- On confirm: studio uses `crypto.getRandomValues` to synthesize per-credential passwords matching the documented complexity rules per credential type, stamps them into the workbook, and triggers two downloads:
  1. `vcf-<version>-<date>.xlsx` — stamped workbook (same as existing export)
  2. `vcf-<version>-<date>-vault.json` — `{ cellAddress: { sheet, label, password, credentialType, complexityRule } }` ready for vault import
- All 5 auto-generate toggles set to `Unselected` in this mode (since the studio is supplying the values).
- Studio memory cleared immediately after; no React state, no module-scoped cache.

**Pros:**
- Solves both Camp A and Camp B in one click.
- Vault file matches the workbook by cell address — auditable.
- Cryptographically strong (`crypto.getRandomValues`, not `Math.random()`).
- No persistence: passwords only exist on the user's disk in the two downloaded files.

**Cons:**
- The user is responsible for handling the vault file (delete after vault import, treat it as Secret).
- If the user re-clicks Export, they get a new set of passwords (no idempotency by design — safer than caching).
- BGP peer passwords still need to be coordinated with the customer's router team after generation.

### Approach C — Hybrid (Recommended)

Combine A's default with B's opt-in:

1. **Default export** (existing button, no new behavior): stamps the 5 auto-generate toggles to `Selected`, leaves all password cells blank. Solves Camp A automatically. Camp B cells get a single comment row in the vault file: "User must supply — see `Plan 13` in README." (Or the studio leaves them blank with a follow-up modal warning.)
2. **Opt-in "Generate Passwords" mode** (new modal accessible from the export bar): user clicks "Generate Passwords for Workbook", picks scope (Camp A only / Camp B only / both), studio generates strong unique passwords, stamps them in, and offers the vault download. The 5 auto-generate toggles are set to `Unselected` for the credentials the studio actually fills.

UX skeleton:
```
[Export Workbook 9.1 (.xlsx)]   ← existing primary button, default Camp A auto-gen
[Generate Passwords…]            ← new modal trigger
  └─ Modal:
     ☐ Generate vault-ready passwords for ALL 70 user-input cells (Camp A + Camp B)
     ☐ Generate ONLY for cells VCF can't auto-generate (Camp B, 18 cells)
     ☐ Skip Camp B BGP peer passwords (I'll coordinate with the router team)
     [Generate & Download Workbook]   [Cancel]
```

**Pros:**
- Sensible default doesn't require any user thought.
- Power-user mode for compliance-driven orgs.
- BGP-aware: lets the user skip generating peer secrets so they don't get downloaded by accident before the router team is briefed.
- Vault file remains opt-in.

**Cons:**
- More code than A or B alone.
- One more modal in the export bar.

**Recommendation: Approach C.** A and B are subsets — if we ship C we can disable the opt-in modal initially and still get A's benefit on day one. Phasing in section 7.

## 4. Security model

Non-negotiables:

### What the studio MUST do

- **`crypto.getRandomValues` only.** Never `Math.random()`. Per-character random selection from the eligible alphabet, not `Math.floor(Math.random() * 64)`.
- **Per-credential entropy.** Every cell gets its own password. No reuse, no derived-from-master scheme. (The "single master + variation" idea adds correlation risk for no real ergonomic gain over the vault file.)
- **Stateless.** No `localStorage`, no `sessionStorage`, no `IndexedDB`, no module-scoped cache, no React `useRef` holding a password. Generated passwords flow from `generatePassword()` → DOM Blob → user disk in a single tick. After the export, the React component's state holds nothing.
- **No clipboard auto-copy.** The vault file gives the user explicit, audited delivery. Clipboard is a side-channel.
- **No URL params, no postMessage, no fetch.** The studio sends nothing over the network. (The shipped HTML loads from CDNs at boot but does no XHR afterward; this property must be preserved.)
- **HTTPS-only context for `crypto.subtle`** if we ever use it; `crypto.getRandomValues` is fine on `file://` per spec.

### What the studio MUST NOT do

- Don't show passwords on screen by default. The vault file is the delivery channel; UI shows counts + cell addresses, not values.
- Don't re-stamp the same passwords on a second export. Each export is a fresh generation — that's the contract.
- Don't add password fields to the JSON export. The fleet design JSON stays credential-free.
- Don't add password fields to the Plan 11 cell-map import side. Importing a stamped workbook does NOT round-trip passwords back into the studio. (The cells exist in the .xlsx; the studio just doesn't read them on import. The vault file is the import surface for secrets.)

### What the user is responsible for

- Storing the downloaded vault file in their secrets manager.
- Deleting the vault file from disk after vault import.
- Coordinating BGP peer passwords with the network team.
- Setting ESX root passwords on the physical hosts before VCF deploy.
- Setting the Encryption Passphrase to a recoverable value (lost passphrase = permanent encryption key loss).

The vault file's top-level JSON should carry these reminders as a header comment so they're impossible to miss.

## 5. Complexity rules per credential type

VCF's per-component complexity requirements (cross-referenced against techdocs.broadcom.com and the workbook's data-validation column lists where present):

| Credential type | Length | Required chars | Notes |
|---|:---:|---|---|
| vCenter root | 8–20 | upper, lower, digit, special | Excludes whitespace, `\`, `'`, `"`, `<`, `>` |
| vCenter SSO admin | 8–20 | upper, lower, digit, special | Same restrictions as root |
| NSX admin / root / audit | 12–128 | upper, lower, digit, special | NSX Manager's stricter min length |
| SDDC Manager root | 12–32 | upper, lower, digit, special | |
| VCF Password (SDDC) | 12–32 | upper, lower, digit, special | |
| Ops admin / root | 12–32 | upper, lower, digit, special | |
| Automation admin | 12–32 | upper, lower, digit, special | |
| Identity Broker admin | 12–32 | upper, lower, digit, special | |
| Edge root / admin | 12–32 | upper, lower, digit, special | |
| ESX root | 7–40 | upper, lower, digit, special | Configurable via host profile; defaulting to 16-char strong |
| BGP peer (MD5) | 8–80 | printable | Must match the peer router's expectations; recommend 24-char alphanumeric |
| Encryption Passphrase | 8–40 | upper, lower, digit, special | Lost passphrase = data loss; recommend 32-char with paper backup |
| vSAN witness root | 7–40 | upper, lower, digit, special | Same restrictions as ESX root |

**Default policy if length/complexity is ambiguous:** 24 characters, 6 each of upper / lower / digit / special, picked from `!@#$%^&*-_=+?` (avoiding `\`, `'`, `"`, `<`, `>`, backtick, semicolon, and Excel-formula triggers `=`, `+`, `-`, `@` at position 0).

A `PASSWORD_POLICY` constant in engine.js encodes these rules:
```js
const PASSWORD_POLICY = {
  "vcenter-root":      { len: 16, classes: { upper: 4, lower: 4, digit: 4, special: 4 }, alphabet: { special: "!@#$%^&*-_=+?" } },
  "nsx-admin":         { len: 24, classes: { upper: 6, lower: 6, digit: 6, special: 6 }, alphabet: { ... } },
  "bgp-peer":          { len: 24, classes: { upper: 6, lower: 6, digit: 12, special: 0 }, alphabet: { ... } },
  ...
};
```

Each `WORKBOOK_CELL_MAP` entry that targets a password cell gains a new field `passwordKind: "vcenter-root" | "nsx-admin" | "bgp-peer" | ...` so the generator knows which policy to apply.

### Vault file format (`vcf-<version>-<date>-vault.json`)

```json
{
  "$comment": "VCF Design Studio generated vault. Save to your password manager and delete this file. Studio retains no copy.",
  "workbookVersion": "9.1",
  "generatedAt": "2026-05-21T16:42:00Z",
  "fleetName": "Acme Production",
  "totalPasswords": 70,
  "credentials": [
    {
      "cellAddress": "Deploy Management Domain!L298",
      "label": "Administrator Password",
      "context": "VCF Operations",
      "credentialType": "ops-admin",
      "password": "Th2X!nQ4mP9-vKsR7uYz",
      "complexityRule": { "len": 24, "classes": { "upper": 6, "lower": 6, "digit": 6, "special": 6 } }
    },
    ...
  ]
}
```

The `$comment` field is JSON-legal (just an unused key); some vault importers strip it on ingest. The `credentials` array is sorted by `(sheet, cellAddress)` so manual cross-check against the .xlsx is trivial.

## 7. Phasing

### Phase 13a — Auto-generate toggle defaults + scaffolding (small)
- Add `WORKBOOK_CELL_MAP` entries (no apply, emit-only) for the 5 auto-generate toggles. Resolver returns `"Selected"`.
- Add `passwordKind` field to the cell-map entry shape (documented, but unused this phase).
- Tests: confirm the 5 toggle cells get stamped `Selected` on default export.
- Coverage: net new ~50 LOC + 6 tests.
- Ship target: 1 small PR.

### Phase 13b — Password policy + generator engine
- New `PASSWORD_POLICY` constant covering ~10 credential types.
- New `generatePassword(kind)` function using `crypto.getRandomValues`.
- New `generateWorkbookVault(fleet, scope)` — walks `WORKBOOK_CELL_MAP` for entries with `passwordKind`, generates per-cell passwords, returns `{ passwords: Map<cellAddress, password>, vault: VaultJson }`.
- 50+ password cells in the cell-map get `passwordKind` fields (no apply — strictly emit-only).
- Tests: complexity-rule satisfaction (every generated password matches its policy), entropy bounds, no `Math.random` (lint rule), no persistence (memory cleared after export).
- No UI changes yet — generator is callable but not exposed.
- Coverage: ~250 LOC engine + ~150 LOC tests.
- Ship target: 1 PR.

### Phase 13c — UI integration
- New "Generate Passwords…" button in the export bar.
- New modal per the mock in section 6.
- New `emitWorkbookXlsxWithPasswords(fleet, fleetResult, pristine, vaultScope)` function in engine.js that walks the cell-map twice: once for the normal Plan 11 cells, once for the password cells using the generated vault.
- Downloads both files (`.xlsx` + `vault.json`) in a single user click; uses `document.createElement('a')` twice with separate Blobs.
- UI E2E test: click the button, complete the modal, verify two file downloads occur and the .xlsx has stamped password values at the expected cells.
- Coverage: ~150 LOC JSX + ~100 LOC engine + ~80 LOC tests.
- Ship target: 1 PR.

### Phase 13d — Documentation + audit trail (optional)
- README section: "Generated passwords + vault delivery"
- Vault file format spec page (for vault-tool integrators)
- Audit-trail comment in vault: `$generator: "vcf-design-studio v9.x"`
- Brief security write-up: "What the studio does and doesn't do with secrets"
- Ship target: docs-only PR.

## 8. Risks

1. **User saves the vault file in their Downloads folder forever.** The studio prints a prominent warning in both the modal and the vault file header, but at the end of the day this is user discipline. Mitigation: make the warning unmissable, recommend automated `~/Downloads/*.vault.json` deletion via a setup script.

2. **Regenerate-on-export overwrites prior passwords.** Once the user has imported the vault into 1Password and deployed VCF, regenerating the workbook will produce *new* passwords that don't match what's actually on the appliances. Mitigation: modal warning, idempotency-via-vault-import feature deferred to a future phase (would require the studio to *read* a vault on import, which crosses the secrets-store line).

3. **BGP peer passwords coordinated with the wrong side.** Generated BGP secrets don't help if the router team isn't told what they are. Mitigation: vault file groups BGP entries together with a comment "Coordinate with network team before applying", and the modal has a "Skip BGP peers" option.

4. **Complexity rule drift between VCF releases.** Broadcom may tighten rules in a future patch; the studio's policy table would need updating. Mitigation: `PASSWORD_POLICY` is version-keyed (matching the `WORKBOOK_CELL_MAP` pattern); workbook updates flow through the cell-meta extractor and a follow-up policy review.

5. **`crypto.getRandomValues` not available in some legacy browsers.** All target browsers (Chrome 86+, Edge 86+, Firefox 78+, Safari 14+) support it; the studio's CDN-loaded React 18 already drops legacy browser support. Mitigation: feature-detect at load, surface a clear error if missing.

6. **Vault file becomes the new attack surface.** A stolen vault file = full fleet credentials. Mitigation: studio doesn't encrypt the vault (we don't have a passphrase to derive a key from; introducing one would be a new credential the user has to remember). Document that the vault file should move to a secrets manager *immediately* and be deleted from disk. A future enhancement could let the user paste a passphrase to encrypt with AES-GCM, but that's Phase 14 if it lands at all.

7. **Excel data-validation complaints.** Several password cells have list-validation drop-downs (e.g. `"VMw@re1!VMw@re1!"` is the default sample). Writing a freshly-generated string passes Excel's pattern check but might trip up downstream VCF Lifecycle Manager validation if the password doesn't match its expectations. Mitigation: Phase 13b's policy table cross-checks against `dataValidation` arrays from the cell-meta fixtures; tests assert generated passwords pass the validators.

8. **Browser memory inspection.** Generated passwords briefly exist in JS heap. A determined attacker with browser-extension privileges can read them. Mitigation: out of scope — same threat applies to every web-based password generator (1Password's web app, Bitwarden, etc.). The studio's job is to be no worse than that baseline.

## 9. Decisions to make before starting

| # | Decision | Default proposed | Notes |
|---|---|---|---|
| 1 | Approach A, B, or C? | **C (hybrid)** | Default auto-gen, opt-in vault generator. A is a strict subset; trivial to ship as the default mode of C. |
| 2 | Phase 13a in isolation, or bundle with 13b? | **Bundle (single PR)** | 13a is too small to justify a standalone PR. |
| 3 | Encrypt the vault file with a user passphrase? | **No (defer to Phase 14)** | Introduces a new credential and a new failure mode (lost passphrase). |
| 4 | Show generated passwords in the UI before download? | **No** | Vault file is the delivery channel. Side-channel-by-default principle. |
| 5 | Auto-fill an existing vault file on re-export to preserve idempotency? | **No** | Reading a vault crosses the secrets-store line and adds complexity. Users who need idempotency can pin a workbook copy and never re-export. |
| 6 | Cover Camp B BGP peer passwords by default? | **Yes, with an opt-out** | Modal exposes "Skip BGP peers" for users coordinating manually with the network team. |
| 7 | Where to gate Camp B SSO Admin / Encryption Passphrase? | **Generate by default** | These ARE strong-secret candidates and the user can override in Excel before deploy if they need a specific value. |
| 8 | Stamp the auto-gen toggles to `Selected` even in the no-password-generate default mode? | **Yes** | The opposite (`Unselected`) is strictly worse — user gets blank password cells with no fallback. |

## 10. Out of scope (intentionally)

- **Reading passwords from the vault on import.** The studio import side (Plan 11 Phase 2) does not consume the vault. Passwords flow studio → vault → vault tool → human; never the reverse through the studio.
- **Per-user / per-fleet password aging.** Vault tools own rotation, not the studio.
- **Active Directory bind password.** Plan 11 already documented that AD sheets (Sheet 26/27) are out of scope until the studio gains an `adConfig` model. Phase 13 doesn't touch AD bind creds.
- **Customer-managed CA signing keys.** The Configure Mgmt Domain sheet has CA-related cells; those are out of scope (operational, not design-time).
- **Encrypted at-rest vault format.** Phase 14 territory if user demand emerges.
